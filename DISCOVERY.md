# Polymarket Wallet Discovery — источники, смещение, архитектура

Документ отвечает на три вопроса: (1) каталог всех источников кошельков; (2) где источники
систематически теряют прибыльных трейдеров (bias) и как это уменьшить; (3) несмещённая
архитектура discovery. В конце — что реализовано в этом проекте.

Помечено: **[проверено]** — сам дёргал эндпоинт в этом проекте; **[известно]** — из общего
знания инфраструктуры, требует разовой проверки перед продакшеном.

---

## ЧАСТЬ 1. Каталог источников

### A. Официальные HTTP API Polymarket

1. **Data API** `data-api.polymarket.com` **[проверено]**
   - Данные: `/activity` (полная лента событий кошелька: TRADE/REDEEM/SPLIT/MERGE/REWARD),
     `/positions` (позиции с realizedPnl/cashPnl/curPrice), `/trades` (глобальная и по рынку,
     несёт proxyWallet+usdcSize), `/holders` (топ-холдеры токена).
   - API: да, без ключа.
   - Ограничения: **offset упирается в потолок ~3500** (история/лента обрезаются); `/holders`
     capped 20 на токен; rate-limit (429) от Cloudflare.
   - Массовость: средняя. Хорош для ДЕТАЛИ по кошельку и для сбора участников по конкретному
     рынку. Плох как источник «всех кошельков» (потолок offset).
   - Дедуп: ключ — `proxyWallet` (lowercase).

2. **Gamma API** `gamma-api.polymarket.com` **[проверено]**
   - Данные: `/markets`, `/events`, `/tags` (3000 тегов), фильтр по `tag_id`, `closed`,
     сортировка по `volumeNum`. Кошельков НЕ даёт — даёт conditionId рынков.
   - API: да, без ключа. Пагинация работает.
   - Массовость: косвенная — карта рынков для последующего сбора участников/холдеров.
     Смещена к крупным/тегированным рынкам.
   - Дедуп: по conditionId на уровне рынков.

3. **CLOB API** `clob.polymarket.com` **[проверено]**
   - Данные: ордербук, `/price`, `/midpoint`, `/midpoints` (батч — текущие цены по токенам),
     история сделок CLOB.
   - API: да. Используем `/midpoints` для оценки открытых позиций.
   - Массовость: рыночные данные, не дискавери кошельков.

4. **Leaderboard API** `lb-api.polymarket.com` **[проверено]**
   - Данные: `/profit`, `/volume`, окна `all`/`30d`/`1d`; поля proxyWallet, amount, name.
   - Ограничения: **потолок топ-50 на запрос, offset НЕ работает.** Сильнейший bias (см. ч.2).
   - Массовость: низкая (≤~300 уникальных со всех срезов). Высокое качество, но узко и смещённо.

### B. Субграфы (The Graph / Goldsky)

5. **Polymarket Orderbook Subgraph (Goldsky)** **[проверено]**
   - Эндпоинт (публичный): `api.goldsky.com/.../polymarket-orderbook-resync/prod/gn`.
   - Данные: сущность `account` с **агрегатами по каждому кошельку**: `scaledProfit`,
     `scaledCollateralVolume`, `numTrades`, `creationTimestamp`, `lastTradedTimestamp`;
     плюс `marketPosition`, `orderFilledEvent`, `condition`, `split/merge/redemption`,
     `marketProfit`. `globals.numTraders` = **1 761 864** — вся популяция.
   - API: GraphQL, без ключа.
   - Ограничения: публичный инстанс **жёстко таймаутит** на `orderBy`/`where` по неиндексированным
     полям и на больших страницах — **макс ~100 строк/стр, только пагинация по `id`**. Профит
     считается по филлам ордербука (может недосчитывать AMM/редимы — это ПРЕ-фильтр, не истина).
   - Массовость: **потенциально максимальная (вся популяция + профит по каждому)**, НО упирается
     в таймауты публичного инстанса → нужна пагинация по id мелкими страницами (часы) ИЛИ
     самохостинг субграфа.
   - Дедуп: `account.id` = адрес (lowercase) — совпадает с proxyWallet в Data API.
   - Прочие субграфы (`-activity`, `-pnl`, `-positions`) на этом проекте отдавали 404 —
     именование меняется, проверять актуальные через дашборды Goldsky/докуи.

### C. On-chain (Polygon)

6. **Polygonscan API** **[известно]**
   - Данные: логи событий и трансферы по контрактам Polymarket на Polygon. Ключевые контракты:
     Conditional Tokens (CTF, Gnosis) `0x4D97DCd97eC945f40cF65F87097ACe5EA0476045`;
     collateral USDC.e `0x2791Bca1f2de4661ED88A30C99A7a9449Aa84174`; CTF Exchange и
     NegRisk Exchange/Adapter (адреса сверить на Polygonscan/докуи). События `OrderFilled`/
     `PositionSplit`/`PositionsMerge`/`PayoutRedemption` → maker/taker = все трейдеры.
   - API: да, бесплатный ключ (5 rps, 100k/день).
   - Ограничения: парсинг ABI/тем логов; пагинация по блокам; 100k/день лимитирует полный охват
     (но за неделю реально стянуть всё). Полностью **несмещённый** срез (вся ончейн-активность).
   - Массовость: максимальная и без selection-bias. Дороже в инжиниринге.
   - Дедуп: адрес maker/taker (lowercase). Учесть: proxy-wallet (Safe) vs EOA-владелец.

7. **RPC + eth_getLogs (собственная нода/Alchemy/Infura/QuickNode)** **[известно]**
   - То же, что Polygonscan, но без дневных лимитов (свой провайдер). Лучший путь для ПОЛНОГО
     несмещённого индекса филлов. Нужен индексер.

### D. Аналитические платформы

8. **Dune Analytics** **[известно]**
   - Данные: spellbook-таблицы Polymarket + сырые `polygon.logs`; готовые дашборды по PnL
     трейдеров. Можно SQL-выгрузить списки кошельков с метриками.
   - API: да, но исполнение запросов — платный/лимитированный тариф; ручной экспорт CSV бесплатно.
   - Массовость: высокая (SQL по всей ончейн-истории). Bias = от автора запроса; сырьё несмещённое.
   - Дедуп: в SQL (DISTINCT trader).

9. **Flipside / Allium / Footprint / Goldsky-Mirror** **[известно]** — аналоги Dune; Flipside
   исторически бесплатный API на SQL. Те же возможности массового экспорта кошельков.

### E. Социальные / репозитории / датасеты

10. **Discord (Polymarket), Twitter/X, Reddit r/Polymarket** **[известно]**
    - Данные: имена/адреса известных шарпов, скрины позиций, «киты». Очень высокое качество,
      малый объём, ручной ввод (seed).
    - API: Twitter/X — платно; Discord — бот в их сервере. В основном ручной сбор.
    - Дедуп: ручная сверка адресов.

11. **GitHub** **[известно]**
    - `py-clob-client`, субграф-репы Polymarket, комьюнити-боты/трекеры; иногда публикуют
      датасеты адресов и готовые ABI/эндпоинты. Полезно для инфраструктуры, не как поток кошельков.

12. **Сторонние трекеры** (polymarketwhales, predictionmarket-трекеры, кастомные лидерборды)
    **[известно]** — экспонируют адреса «китов»/прибыльных; bias как у лидерборда.

13. **Академия / Kaggle / HuggingFace** **[известно]** — статьи об эффективности prediction
    markets, иногда с выгрузками; единоразовые датасеты, устаревают.

## ЧАСТЬ 2. Где источники СИСТЕМАТИЧЕСКИ теряют прибыльных трейдеров (bias)

Главная мысль: почти все «удобные» источники — это **выборка по нашему запросу или по чужому
рейтингу**, а не по популяции. Отсюда систематические дыры:

1. **Bias лидерборда (размер + выживший + окно).** Топ-50 по абсолютному $ → только киты и
   маркетмейкеры; небольшой, но процентно-сильный кошелёк не попадёт никогда. Окна all/30d/1d
   режут «сезонных» (политика раз в цикл). Митигейт: не опираться на лидерборд как на основу.

2. **Bias объёма рынка.** Свип «топ-рынки по volume» = выборы/мега-события → **категорийный
   перекос в политику** и доминирование MM в этих рынках. Средние/нишевые рынки и их специалисты
   выпадают. Митигейт: стратификация по децилям объёма рынка, а не только топ.

3. **Bias со-трейдеров по объёму.** Ранжирование участников рынка по $ снова вытаскивает
   маркетмейкеров (высокий объём, нулевой эдж). **[проверено: в спорте 55→только 6 с ci_low>0]**.
   Митигейт: ранжировать со-трейдеров по **частоте со-появления** (специализация), не по объёму.

4. **Bias «только активные».** Фильтр live≤14д выкидывает сильных, кто ждёт своего события
   (политструктуры, сезонщики). Митигейт: для discovery держать окно шире (90–180д), live
   применять только на этапе ВКЛЮЧЕНИЯ в копи, не на этапе поиска.

5. **Bias холдеров.** `/holders` cap 20 на токен → только крупнейшие держатели = размерный перекос.

6. **Bias «только закрытые рынки».** Анализ по resolved пропускает специалистов долгих открытых
   позиций. Митигейт: включать активные рынки в карту.

7. **Bias offset-потолка (3500).** Data API режет историю высокочастотников → их метрики искажены
   (выглядят хуже/иначе). Митигейт: для таких брать ончейн/субграф, а не activity.

8. **Bias пре-фильтров (min-vol/min-stake).** Отсечка по объёму/стейку режет **мелких чистых
   шарпов** — а их копировать как раз легче (не двигают цену). Митигейт: низкие пороги на этапе
   discovery, отбор уже по эджу.

9. **Bias тегов.** Поиск через теги Gamma теряет рынки без тегов и новые темы. Митигейт: дополнять
   ончейн-перечислением, не зависящим от тегов.

10. **Bias survivorship/recency в самих метриках.** Малый n + бутстрап: один джекпот завышает ROI.
    Уже лечим bootstrap-CI (ci_low) и resolved_pnl.

**Вывод:** единственный способ убрать selection-bias — начать с **полной популяции (ончейн/субграф)**
и далее СЕМПЛИРОВАТЬ её осознанно, а лидерборд/объём/со-трейдеры использовать лишь как
augmentation поверх, помечая провенанс.

## ЧАСТЬ 3. Несмещённая архитектура discovery

Цель — репрезентативный пул, а не быстрый. Слои:

**Слой 0. Популяционный фрейм (спина).**
Перечислить ВСЕ кошельки из ончейна/субграфа (`account` или `OrderFilled` логи). Это
1.76M адресов с агрегатами (профит, объём, n_trades, last_traded). Реализация: либо самохостинг
субграфа / свой RPC-индексер (без таймаутов), либо Dune SQL, либо мелкостраничная пагинация
публичного Goldsky по `id`. На выходе — таблица `wallet, profit, volume, n_trades, first/last_ts`.

**Слой 1. Стратифицированный семпл (анти-bias).**
Из фрейма не брать «топ по X», а семплировать по стратам, чтобы не перекосить:
- по **размеру стейка** (децили): включать и мелких, и крупных;
- по **категории** рынков, где торгует кошелёк (политика/крипто/спорт/...): квоты на категорию;
- по **давности/частоте**: и активные, и сезонные;
- по **возрасту аккаунта**: и ветераны, и новички с сильным стартом.
Внутри страты ранжировать по грубому профиту (субграф) и брать верх — это даёт сильных
**малоизвестных** игроков, которых нет в лидерборде.

**Слой 2. Таргетированное augmentation (известно-хорошее, но смещённое — помечаем провенанс).**
- Лидерборд (profit/volume × all/30d/1d).
- Со-трейдеры наших победителей **по частоте** (специалисты их полей).
- Холдеры резолвнутых рынков с высоким эджем.
- Ручные seed из Discord/Twitter.

**Слой 3. Единый отбор.**
Один и тот же строгий анализатор (bootstrap-CI эджа, resolved_pnl, не-MM, не-один-колл) гоняется
по ВСЕМУ объединённому пулу. Поскольку фрейм = популяция, отбор несмещён.

**Дедуп и провенанс.**
- Канонический ключ: `lower(proxyWallet)` — совпадает между Data API, субграфом (`account.id`),
  ончейн-логами (maker/taker).
- Master-таблица: union (не intersection), с полем `sources: set` (провенанс каждого).
- Нюанс: один человек = несколько proxy/Safe; полностью свести к человеку нельзя и не нужно —
  торговая идентичность (proxy) и есть единица копирования.
- Хранить «первый увидевший источник» для атрибуции эффективности каналов (у нас уже есть
  `wallet_sources.json` + колонка «Источник» на дашборде).

**Порядок построения базы:** Слой 0 (популяция) → Слой 2 (быстро добавить известных) →
Слой 1 (стратифицированный добор из популяции) → Слой 3 (анализ всех) → форвард-валидация.

## ЧАСТЬ 4. Что уже есть в проекте и что строю по этому документу

Есть: data-api/gamma/clob/leaderboard конвейеры; co-traders **по частоте**; нишевые/категорийные
теги (политика/крипто/финансы/...); булк-харвест уже проанализированных; `wallet_sources.json`
(провенанс) + колонка на дашборде; форвард-валидация (реализ.+нереализ. PnL, флаги).

Строю сейчас (Слой 0, несмещённый): **`subgraph_enum.py`** — мелкостраничная (100/стр) пагинация
субграфа по `id`, сбор кошельков с `scaledProfit` выше порога и `lastTradedTimestamp` в окне,
с чекпоинтом курсора и ретраями. Это даёт случайный по id (=несмещённый) срез популяции,
отфильтрованный по грубому профиту, → затем в общий анализатор.

Дальнейшее (рекомендации, требуют ресурса/решения): свой RPC-индексер `OrderFilled` или Dune SQL
для ПОЛНОГО фрейма без таймаутов; стратифицированный семпл по категориям/децилям.
